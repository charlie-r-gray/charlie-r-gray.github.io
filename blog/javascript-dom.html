HTML & CSS

You know about HTML and CSS, though perhaps not by those names.

HTML is the content of the webpage you see: the words, the logos, the links. It allows you to enter and submit your own content, too, in the case of forms.

CSS is the flair of the webpage you see: the colours, placement, animation effects. If can also change the way pages look at certain sizes, or even prevent you from resizing them below certain thresholds, remove scrolls bars, pop out menus, and more.

You can think of HTML as the book's fairy tale, and CSS as the book's cardstock and fancy Ye Olde Gothick giant capital letters.

Control flow, and loops

Control Flow

Control Flow determines the order in which script will run. Most code runs top to bottom, but each type of code we use in website design (JavaScript, CSS, and HTML) have their own control flow.

HTML will exclusively read from top to bottom. This means if you want scripts to do something to your page, you must add below the HTML they are intended to execute upon. If you want to use resources in your HTML (such as fonts), you must tell the HTML where they are before it tries to use them. HTML doesn't want to look ahead - if it can't do it right away, it won't do it at all.

CSS will generally favour top to bottom, but: #ids take priority over .class, and both take priority over type (eg: h1, p), regardless of their position relative to each other in the code. Anything within @media queries will take priority over anything outside, as long as those conditions are met. CSS assumes your more recent changes are typically what you want, and will 'overwrite' previous ones - but it still understands that some things are more important! than others.

JavaScript still needs you to declare variables before you use them (top to bottom), but function order is a matter of readability. Functions don't do anything one their own when declared (written); they must be called. When a function is called, the script will look for them anywhere within the script contents, even if the function is below the line calling it.

Loops

Sometimes when we're writing code, we want our code to do something long, complicated, and do it several times. We may also need that code to be applied to information we might not have yet - such as something a website user tells us. For that, we need to use loops.

Loops are how we tell our code that we want them to do the same thing over and over, a set number of times. We use variables to make that useful - starting with one to track how much progress the loop has made (often using i, for 'iterator'). We also use variables to track what the loop is actually doing - maybe we pass it an array and use i to get it to modify each part of the array. Or perhaps we pass in some information, but do something to it i times.

The DOM

'Document Object Model'. It's a technical name, but it's just way to refer to all the website stuff, and how the stuff all relate to each other, and are all inside the other things.

Okay, uhh.. maybe that's a bit too simplistic?

Alright, so when you're looking at a website, you're using a browser, right? You typically call that a 'window'. Minimise your window, resize it, whatever. The stuff that's in your window is your document - that is, the whole website. But the website is still made of a bunch of stuff, right? It's pictures, HTML, CSS, JavaScript. All those things are part of the website. But each of those contain their own things, too! HTML has the head (like the page title), the body (the words in your page), your footer with the links along the bottom. Yeah that's right, even the footer has more stuff inside it - it's stuff all the way down.

Arrays and Objects

Arrays and objects are both ways to store multiple kinds of information. A number is always just a single number. A float is a more complex number, but still just the one. And I could paste the entirety of Lorem ipsum to make a very long string, but still just the one string. Arrays and objects though? They like to juggle a lot of bits.

Arrays

Arrays are your choice for when you want to hold a bunch of information in a ordered, but unlabelled, fashion. Arrays can hold numbers, strings, floats, perhaps other variables - perhaps even a mix of all of those. However, they're basically stored as a simple list. Though arrays can hold a lot of information, they're not labelled. You can't use dot notation to search for anything specific, instead have to use index numbers (that is, the order they're stored in!). They're the best choice for holding a lot of very similar information: such as a list of client names. And they're best used for loops where the whole array will be checked.

[lime, lime, lime, lime, unripe lemon, lime, lime]

Objects

Objects can hold all the limes, and all the pineapples, mangoes, and kiwifruit too - and they know which is which. Objects can hold almost anything, including other objects. And they have a way to tell what is what. Unlike array, that pours all its limes on the counter one after another, an object can-- look, this metaphor is starting to fall apart.

Let's change it up: an array may be good for remembering your client names, but how much good does that do you by itself? You also want to know a client's phone number, their eMail address, and, most important - how to tell which of those is which.

client0451 = {
  name:"Lime Guy",
  phone:555-5463,
  email:"luv-limes@yahoo.io"
}

This means you can use dot notation to call out something specific inside the object, such as a phone number: client0451.phone will return the number "555-5463".

By their powers combined...

Objects can hold arrays, and vice versa. So we can make an array for clients, have each entry in that array be an Object that contains each client's name and contact details.

Functions

Functions are the core of what makes JavaScript useful - it's the core of JavaScripts actions. You can certainly have JavaScript do a bunch of wizardry the first time it's loaded in the HTML, but if you want it to keep doing things, or react the user, you need functions.

Thing of functions as like assembly instructions. On its own, a function doesn't do anything. It's an inert machine, waiting to fed material. Say... a machine that slaps a packing label on a box. But if you tell javascript to read those instructions (call the function), it will go through those instructions. Click a button, and javascript will slide a box into the function, and get a label slapped onto it ready for shipping.

Functions also have parameters (the content of the brackets). On their own they don't mean anything, it's just a way for a function to refer to what it's going to be doing it's thang to. Think of it as a function-exclusive variable that is declared each time the function is called.

Lets give our boxLabel() one with boxLabel(name) - now when we call that function, our packing label will include our addressee's name. But how does this machine know the name to put on this label? It's passed, when called: "Hey machine, print me a boxLabel("Martha White")." Now when the function prints its label, and it reaches boxLabel(name), it knows that name = "Martha White".
